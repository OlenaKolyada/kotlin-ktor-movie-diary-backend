# Vulpecula

[English](README.md) | Русский

**Vulpecula** — учебный backend-проект на Kotlin/Ktor: RPC-style HTTP API для дневника просмотров фильмов. Пользователи могут вести историю просмотров, сохранять оценки и комментарии к фильмам, формировать личную коллекцию просмотров и получать статистику по своим предпочтениям. Проект выполнен в рамках курса **Kotlin Backend Developer**, поток **2025-08**. Проект назван в честь созвездия Vulpecula (лат. "Лисичка"), в память о любимой собаке корги, которую ласково называли "лисичкой".

## Кратко о проекте

**Vulpecula** показывает разработку high-load backend-приложения на Kotlin с модульной архитектурой, OpenAPI-контрактом, разделением слоёв, тестированием и подготовкой к контейнеризации.

Проект не является production-сервисом. Это учебный проект, цель которого — показать подход к проектированию high-load backend-приложения, структуре кода и документации.

## Формат API

Vulpecula использует RPC-style HTTP API: операции представлены как action-oriented POST endpoints, а не как классические REST-ресурсы. Такой формат API задан архитектурными требованиями курса.

## Что демонстрирует проект

- Backend-разработку на Kotlin и Ktor
- Проектирование HTTP API и OpenAPI-контракта
- Построение приложения по принципам Clean Architecture
- Разделение бизнес-логики, транспортного слоя и инфраструктуры
- Framework-agnostic подход: бизнес-логика не зависит от Ktor
- Database-agnostic подход: доменная и бизнес-логика не завязаны на конкретную базу данных
- Разделение транспортных, доменных и бизнес-моделей
- Маппинг между слоями приложения
- Модульную структуру Gradle-проекта
- Unit-тестирование сериализации, мапперов и бизнес-логики
- Подготовку к e2e-тестированию и контейнеризации
- Интеграцию с брокером сообщений
- Подготовку к работе с NoSQL-хранилищем

## Основные возможности

- Создание записи о просмотре фильма
- Чтение записи дневника
- Обновление оценки, комментария и данных просмотра
- Удаление записи
- Работа со стабами для тестирования бизнес-сценариев
- Описание API через OpenAPI-спецификацию
- Генерация транспортных моделей по контракту

## Архитектура

Проект разделён на модули по зонам ответственности:

- API и транспортные модели
- Общие доменные модели
- Бизнес-логика
- Репозиторий
- Мапперы между слоями
- Стабы для тестирования сценариев

Бизнес-логика отделена от Ktor, транспортных моделей и конкретной реализации хранения данных. Это позволяет менять внешний API, фреймворк или базу данных с меньшим влиянием на ядро приложения.

## Статус проекта

Проект находится в разработке как учебный backend-проект. Часть функциональности реализована в коде, часть описана в документации как проектные требования, архитектурные решения и планируемые сценарии.
